## OpenDB 是什么？

<h3>OpenDB 是一个数据库 CLI Agent — 参考Claude Code交互方式的数据库智能体。用最简洁的交互，实现最优的管理和诊断。</h3>

<h3>当前支持 Oracle、MySQL、PostgreSQL 三大主流数据库。同时支持三种交互方式，并把它们完美整合到了一个 Agent 智能体中：</h3>

<h3>OpenDB 自动识别输入类型并路由到对应处理器 — 无需切换模式</h3>

<h3>（1）/ 简化命令（类似 Claude Code）</h3>

把常用的复杂运维操作整合到一个简单的命令里，一键完成：

```
opendb> /health                     ← 一键全维度健康体检（20+ 项）
opendb> /dbtop                      ← 实时性能面板，类似 Linux top
opendb> /slowsql 500                ← 查找超过 500ms 的慢 SQL
opendb> /sentinel                   ← 启动后台实时异常探针
opendb> /rule                       ← 规则引擎一键诊断
opendb> /llm                        ← AI 大模型深度诊断
opendb> /kill 1842                  ← 终止问题会话
opendb> /explain a3f8k2j9x          ← 查看 SQL 执行计划
```

<h3>（2）SQL 和数据库特有语法（类似 sqlplus / mysql / psql）</h3>

直接输入 SQL 或数据库原生命令，无缝兼容你已有的操作习惯：

```
opendb> SELECT sid, event, seconds_in_wait FROM v$session WHERE status = 'ACTIVE';
opendb> ALTER SYSTEM KILL SESSION '1842,30721' IMMEDIATE;
opendb> SHOW PARAMETER sga_target;
opendb> DESC dba_tablespaces;
```

<h3>（3）自然语言（类似聊天工具）</h3>

直接用自然语言描述你的问题，OpenDB 自动路由到 AI Agent 进行智能响应：

```
opendb> 数据库现在为什么慢？
⚡ 分析活跃会话中... 42 个会话等待 "enq: TX - row lock contention"
📋 根因：会话 SID=1842 在 ORDER_MASTER 上持有行锁已 47 分钟
🔧 建议操作：
   ALTER SYSTEM KILL SESSION '1842,30721' IMMEDIATE;

opendb> 帮我看看今天最慢的5条SQL
opendb> 表空间快满了怎么处理
opendb> 最近有没有 ORA 报错
```
### 三种交互方式，合多为一

OpenDB 自动识别输入类型并路由到对应处理器 — 无需切换模式：

| 模式 | 示例 | 执行效果 |
|------|------|---------|
| **/ 命令** | `/slowsql 500` | 执行内置技能：查找超过 500ms 的慢 SQL |
| **原生 SQL** | `SELECT * FROM v$session` | 直接在数据库上执行，等同于 sqlplus/mysql/psql |
| **自然语言** | `帮我看看今天最慢的5条SQL` | 路由到 LLM Agent 进行智能响应 |

## 快速开始

**安装**（Linux / macOS）：

```bash
curl -fsSL https://opendb.ai/install.sh | bash
```

**运行安装向导：**

```bash
opendb setup
```

向导会引导你完成数据库连接、LLM 配置和规则引擎设置 — 一分钟内即可启动。

**修改已有配置：**

```bash
opendb configure
```

随时调整数据库连接、LLM 模型或规则引擎等配置项。

## 特色功能

| 功能 | 定位 | 一句话介绍 |
|------|------|-----------|
| [`/health`](#health--全维度健康体检) | 全局体检 | 20+ 项指标一键扫描，7 大维度（实例/空间/会话/内存/日志/备份/安全）给出 ✓/⚠/✗ 状态 |
| [`/dbtop`](#dbtop--实时性能面板) | 秒级快照 | 类似 Linux `top`，1 秒刷新 SGA/PGA/TPS/QPS/等待事件/活跃会话，捕捉瞬间状态 |
| [`/sentinel`](#sentinel--实时异常检测) | 持续监控 | 后台探针 48 项指标 × 9 种检测策略，自适应 3σ 基线，异常自动 Burst 采集证据 |
| [`/scheduler`](#schedule--自动化巡检) | 定时巡检 | 将 health / 报告 / 维护整合为 cron 定时任务，后台自动执行 |
| [`/rule`](#rule--确定性决策引擎) | 规则诊断 | 273 条规则 × 5 阶段管线，毫秒级确定性诊断，离线可用，无需 LLM |
| [`/llm`](#llm--大模型智能诊断) | AI 诊断 | 大模型 function calling 驱动最多 20 轮链式推理，自动采集证据，收敛到根因 |

### `/scheduler` — 自动化巡检

将日常巡检、定期报告和例行维护整合到后台定时任务中 — 不需要 crontab，不需要脚本，不需要来回切换。

```
opendb> /scheduler start health-check --cron "0 8 * * *"
✓ 已调度: health-check 每天 08:00 执行
```

### `/llm` — 大模型智能诊断

OpenDB 使用大模型的 function calling 驱动多轮、基于证据的诊断。Agent 不靠猜 — 它查询数据库、分析结果、形成假设、反复迭代，直到定位根因，并给出可直接执行的修复 SQL。

### `/rule` — 确定性决策引擎

当 LLM 不可用或需要极速响应时，Rule 引擎提供毫秒级确定性诊断。这不是简单的 if-else 规则 — 它是基于 Claude Opus 专家级 DBA 诊断能力提炼的**多阶段决策管线**，包含 273 条专用规则。

**真实压测场景下的规则诊断输出（2.7 秒完成）：**

```
opendb> /rule

── 规则诊断 ────────────────────────────────────────────────────

  根因: 序列 CACHE 过小或 NOCACHE 导致 SQ 争用
  严重程度: ■■■□ 高    置信度: 78%

  ── 证据链 ──
  1. enq: SQ - contention 在单实例环境下通常由序列 CACHE 值过小引起
  2. 每次 CACHE 用完需要更新数据字典 seq$，引发 row cache lock 和 SQ enqueue 争用
  3. 高并发 INSERT 使用 NOCACHE 或小 CACHE 序列时尤为严重
  4. 关联规则 WE2-007 (enq: SQ — 序列跨实例排序争用诊断 (RAC)) 建议一并排查
  5. 关联规则 WD015 (row cache lock 诊断) 建议一并排查

  ── 处置建议 ──
  [紧急] 1. 查看争用序列并增大 CACHE
         > /sql "SELECT s.sequence_owner, s.sequence_name, s.cache_size,
           s.order_flag, s.last_number FROM dba_sequences s
           WHERE s.sequence_owner NOT IN ('SYS','SYSTEM','AUDSYS','DBSNMP')
           ORDER BY s.cache_size ASC FETCH FIRST 20 ROWS ONLY"
         风险: 序列号间隙增大（不影响唯一性）
  [修复] 2. 查看 ASH 确认争用序列对象
         > /sql "SELECT h.current_obj#, o.object_name, COUNT(*) waits
           FROM v$active_session_history h LEFT JOIN dba_objects o
           ON h.current_obj# = o.object_id
           WHERE h.event = 'enq: SQ - contention'
           AND h.sample_time > SYSDATE - 1/24
           GROUP BY h.current_obj#, o.object_name
           ORDER BY waits DESC FETCH FIRST 10 ROWS ONLY"
         风险: 无
  [预防] 3. 对高频 INSERT 表考虑用 UUID 或应用层生成 ID 替代序列
         风险: 需要应用改造

  ── 关联分析 ──
  • cursor: pin S 诊断: 受根因影响的下游症状，解决根因后将同步缓解
  • AWR Segment Statistics 热点对象定位: 受根因影响的下游症状，解决根因后将同步缓解
  • buffer busy waits 诊断: 受根因影响的下游症状，解决根因后将同步缓解

──────────────────────────────────────────────────────────────────
```

### Rule — 多阶段决策管线

**关键技术难点：**

- **5 路倒排索引**：按等待事件、错误码、指标名、分类标签、关键词建立 O(1) 倒排索引，从 273 条规则中毫秒级定位候选规则，而非逐条遍历
- **决策树嵌套**：单条规则最深支持 20 层嵌套判断、50+ 分支条件。例如 `WE007`（行锁争用）的决策树需依次判断：是否死锁 → 是否 DDL 阻塞 → 阻塞者是否空闲 → TX 锁模式 → 长事务检测，每层通过 32 条预定义 SQL 实时采集证据
- **因果链消解**：当多条规则同时命中时（如 SGA 耗尽同时引发 Buffer Hit 下降 + I/O 等待增加），通过 CausesOf/CausedBy 有向图自动识别根因 vs 下游症状，用评分公式 `Score = SeverityWeight × Confidence × Specificity` 收敛到 1-2 个核心原因

### LLM — 多轮链式推理引擎

LLM Agent 不是一次性问答 — 它是一个**可控的多轮推理循环**，每轮通过 function calling 调用 30+ 个数据库查询工具采集证据，逐步收敛到根因：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    LLM Agent 推理循环                                   │
│                                                                         │
│  ┌─────────────────────┐                                               │
│  │ BurstReport 压缩    │  原始遥测数据 → ~2000 token 结构化摘要        │
│  │ ├─ 指标: db%, wait% │  包含: 分类结果、指标、等待分布、             │
│  │ ├─ Top 等待事件 ×5  │        Top SQL ×3、阻塞链 ×3                  │
│  │ └─ Top SQL ×3       │                                               │
│  └──────────┬──────────┘                                               │
│             │                                                           │
│             ▼                                                           │
│  ┌─────────────────────────────────────────────────┐                   │
│  │              推理循环 (最多 20 轮)               │                   │
│  │                                                  │                   │
│  │  for turn = 0; turn < maxTurns; turn++           │                   │
│  │    ┌──────────────────────────────────────┐      │                   │
│  │    │ 1. 注入收敛提示 (倒数第 2 轮)        │      │                   │
│  │    │ 2. LLM 推理 → 选择工具调用           │      │                   │
│  │    │ 3. 无工具调用 → 已收敛，输出结论     │      │                   │
│  │    │ 4. 执行工具 → 安全校验 → 结果回传    │      │                   │
│  │    │ 5. 证据累积到上下文                   │      │                   │
│  │    └──────────────────────────────────────┘      │                   │
│  │                                                  │                   │
│  │  收敛保障:                                       │                   │
│  │  ├─ 倒数 2 轮: "请直接给出最终诊断总结"         │                   │
│  │  ├─ 超时回退: Auto(10轮) → Assist(3轮)          │                   │
│  │  └─ 最终轮: "这是最后一轮，必须立即给出诊断"    │                   │
│  └─────────────────────┬───────────────────────────┘                   │
│                        │                                                │
│                        ▼                                                │
│  ┌─────────────────────────────────────────────────┐                   │
│  │ 诊断报告                                        │                   │
│  │ ├─ 根因分析 + 证据链                            │                   │
│  │ ├─ 可执行修复 SQL                               │                   │
│  │ └─ 推理链 (每轮思考过程) + Token 统计           │                   │
│  └─────────────────────────────────────────────────┘                   │
│                                                                         │
│  防幻觉机制:                                                           │
│  ├─ 结论中引用的数值必须来自工具查询结果                               │
│  ├─ 没查过的指标不能写进结论                                           │
│  ├─ 修复 SQL 前必须先用 sql 技能验证对象是否存在                       │
│  └─ 危险操作 (ALTER/DROP) 需安全确认                                   │
└─────────────────────────────────────────────────────────────────────────┘
```

**关键技术难点：**

- **收敛控制**：LLM 推理天然具有发散性。OpenDB 通过三重机制保障收敛 — 倒数第 2 轮注入收敛提示、超时自动降级（Auto 10 轮 → Assist 3 轮策略回退）、最终轮强制输出。避免 LLM 无限调用工具却不给结论
- **证据压缩**：Sentinel 的 BurstReport 包含 ~150 帧 × 48 指标的原始数据，直接输入会爆 context。OpenDB 将其压缩为 ~2000 token 的结构化摘要（指标 + 等待分布 + Top SQL + 阻塞链），既保留诊断所需的关键信息，又控制 token 成本
- **防幻觉**：LLM 诊断的最大风险是编造不存在的问题。OpenDB 在 system prompt 中硬性约束「结论中引用的数值必须来自工具查询结果」「修复 SQL 前必须先验证对象存在」，将诊断从"猜测"变为"基于证据的推理"
- **双模型策略**：小模型（≤9B）使用 GuidedStrategy（OpenDB 编排，3 轮只读），大模型（≥27B）使用 AutonomousStrategy（10 轮全工具自主诊断），根据模型能力自动选择最优策略
